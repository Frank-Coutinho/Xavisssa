# Xavissa Licensing and Device Activation Flow

Xavissa treats login and licensing as separate checks.

Authentication proves who the user is. Licensing proves the tenant and this device are allowed to open the real workspace. Valid credentials alone must never open the POS workspace when the tenant license or current device activation is invalid.

## Ownership Boundary

Licenses are generated by the separate Xavissa web backoffice and its controlled backend. The desktop app and local backend do not generate raw license keys, `LicenseKeyHash`, or `LicenseKeyPrefix`, and they do not contain the server secret used to hash license keys.

The desktop app only:

- collects the raw key during first activation or reactivation
- sends the raw key to the configured online licensing API over HTTPS
- receives a server-confirmed activation and signed license snapshot
- stores the signed snapshot locally
- validates that snapshot before opening, syncing, or writing restricted data

The raw key is not stored locally after successful activation.

## Startup

1. The local backend starts as before.
2. Local SQLite schema is initialized.
3. `IDeviceIdentityService` loads or creates the persisted local device id and stable fingerprint.
4. `ILicenseStateService` loads the local signed license snapshot.
5. If no snapshot exists, Xavissa shows the activation/demo screen.
6. If a snapshot exists, `ILicenseSnapshotVerifier` checks the signature, tenant, device fingerprint, activation id, status, expiry, snapshot expiry, grace period, and offline window.
7. When online, Xavissa calls the online licensing API to validate the activation and refresh the snapshot.
8. When offline, Xavissa allows use only while the signed snapshot is valid and within `MaxOfflineDays`.
9. Expired, suspended, deactivated, wrong-device, invalid-signature, or offline-limit failures open a blocked/restricted screen.

## Login

1. The user enters credentials.
2. Xavissa authenticates online or offline using the existing identity cache.
3. The app resolves the selected tenant/store context.
4. `ILicenseStateService` rechecks the license for that tenant and current device.
5. If valid, the workspace opens.
6. If the user is a Clerk or Store Manager and the device is not activated, Xavissa blocks access and tells them to contact the Tenant Admin or support.
7. If the user is Tenant Admin and the device needs activation, they can enter the raw license key.

## Activation

The same raw tenant license key is used on every allowed device. Each device consumes one active activation slot on the licensing server.

During activation, Xavissa sends:

- raw license key
- tenant id/code when known
- user context when known
- device fingerprint and device details
- app and OS version

The licensing API is responsible for the authoritative device limit transaction: lock/check license, count active activations, insert the activation when capacity remains, and return a signed snapshot. Xavissa never relies on local device counting for `MaxDevices`.

## Local Snapshot

The local snapshot contains tenant, license, plan, activation, device, feature flag, expiry, offline, and signature data. It is stored locally and treated as untrusted until verified.

Production verification should use asymmetric signatures with a public key configured as `Licensing:LicensePublicKey`. A temporary unsigned snapshot escape hatch exists only behind `Licensing:AllowTemporaryUnsignedSnapshots` and is disabled by default.

## Demo and Trial

Demo mode is disposable and starts through the licensing/demo API without asking the user to type a license key. The server creates or assigns the demo tenant/session/license and returns a signed demo snapshot. Demo expiry blocks the demo workspace.

Trial mode uses real tenant data with `IsTrial = true` and an expiry. Trial data is not reset automatically and can be converted to paid use by the backoffice.

## Feature and Quota Enforcement

Xavissa does not limit product count, category count, variant count, stock movements, sales volume, or sales count by license.

Feature/quota guards enforce:

- store creation: `MaxStores` and `AllowsMultiStore`
- tenant user creation: `MaxUsers`
- cloud sync: `AllowsCloudSync`
- advanced reports: `AllowsAdvancedReports`
- barcode generation/printing: `AllowsBarcodePrinting`
- custom receipt settings: `AllowsCustomReceipt`
- demo mode: `AllowsDemoMode`
- offline use: `MaxOfflineDays`

UI hides or blocks restricted features where practical, and services still check before sensitive operations.

## Sync

Remote sync first checks the license state. If online validation fails because the license/device is invalid, sync stops. If offline, Xavissa can continue local offline work only inside the signed snapshot offline window, but it does not call remote sync. If `AllowsCloudSync` is false, remote sync is disabled even when the app is online.

Future controlled remote sync APIs should also reject requests from invalid or inactive activations and should receive tenant id, store id, activation/device fingerprint, app version, and snapshot/version context.
